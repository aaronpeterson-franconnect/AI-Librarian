# AI Librarian — Glossary

Terms used consistently across the architecture, ADRs, and (eventually)
the code. When in doubt, defer to this list.

| Term | Meaning |
|---|---|
| **Source** | An immutable raw document submitted to the system: a PDF, a Word doc, a SQL file, a video file, etc. Stored in Azure Blob, fingerprinted by SHA-256. The system never modifies a source. |
| **Chunk** | A canonicalized, retrievable fragment derived from a source. Chunks are produced by the Skill plugin appropriate for the source's format. Each chunk has a span anchor that points back into the source. |
| **Span anchor** | A structured pointer back into the source: page number for PDFs, paragraph index for prose, timestamp for audio/video, line range for code, changeSet ID for Liquibase SQL. The anchor is what makes a citation precise. |
| **Embedding** | The vector representation of a chunk, stored in `pgvector`. Used for semantic search. |
| **Wiki** | The LLM-authored, compounding synthesis of everything the company knows in a given department. Lives in Postgres tables. Humans never edit it directly. |
| **Wiki page** | A logical page in the wiki — for example, an entity page (`Stripe`), a concept page (`Secret Rotation`), a topic summary, or a synthesis. Has a stable ID and slug. A page contains one or more facets; readers see the highest-classification facet they can access. |
| **Page facet** | A content variant of a wiki page keyed by `min_classification`. A page about "Customer onboarding" might have an `Internal` facet (general process, broadly readable) and a `Confidential` facet (specific dollar thresholds, department-only). The Wiki Maintainer produces facets per visibility tier when source material supports them; citations within a facet only reference sources visible at that facet's level. See [ADR 0006](adr/0006-llm-only-wiki-with-directives.md) and [ADR 0011](adr/0011-data-classification.md). |
| **Wiki revision** | A point-in-time snapshot of a page facet. Append-only; the facet's "current" content is the latest revision. Facets within a page have independent revision histories but are always regenerated together when the source pool changes. |
| **Claim** | An atomic factual statement on a wiki page (one row in `wiki_claims`). Immutable once written. Every claim must cite at least one chunk via `wiki_claim_citations`. |
| **Citation** | The link from a claim to a chunk. Includes the chunk ID and (optionally) a more specific span within the chunk. |
| **Skill** | A self-contained file-format plugin (e.g., `Skills.Pdf`, `Skills.Code`). Contains a `SKILL.md` manifest declaring supported MIME types, capabilities, and the canonicalization contract. See [ADR 0009](adr/0009-skill-plugin-pattern.md). |
| **Department** | A scoped corpus with its own librarian(s), policy, and directives. Departments are flat — no parent/child structure. Each `(department, role)` tuple maps to one Entra ID group. |
| **Department policy** | A versioned configuration per department: in-scope topics, out-of-scope topics, quality threshold, approval rules, retention period. Edited by librarians; consulted by the ingestion editorial filter. |
| **Directive** | A persistent content-level instruction that the Wiki Maintainer must obey on every regeneration of pages in scope. The librarian's primary lever for steering wiki content without editing pages directly. |
| **Page lock** | A flag on a wiki page that forces any maintainer-proposed update into the approval queue rather than committing automatically. Used sparingly for high-stakes pages. |
| **Approval queue** | The list of items waiting on a librarian's decision: pending sources, locked-page proposed revisions, contested classification calls. |
| **Librarian** | A user with the `Librarian` role on one or more departments. Approves/rejects sources, edits policy and directives, marks pages as locked, initiates concept-level RTBF reviews. The "human in the loop." |
| **Reader / Contributor / Reviewer / Admin** | Other roles. Reader: query-only. Contributor: can submit sources. Reviewer: approve/reject in the queue and soft-delete, but cannot edit policy or directives. Admin: system-wide configuration, can create departments and assign librarians. |
| **Reviewer** | A user with `Reviewer` rights on one or more departments. Approves and rejects items in the approval queue and may soft-delete sources. Does not edit policy, directives, or page locks; cannot hard-delete. Separation-of-duties role between Contributor and Librarian. |
| **MCP** | Model Context Protocol — the Anthropic-originated protocol for connecting AI clients to tool-providing servers. We expose one MCP server as the single AI access layer. See [ADR 0004](adr/0004-mcp-as-single-access-layer.md). |
| **Wiki Maintainer** | The background agent that integrates approved sources into the wiki. Writes claims and revisions; respects directives and locks. |
| **Linter** | The background agent that periodically checks the wiki for contradictions, stale claims, orphans, missing citations, and coverage gaps. Suggests; does not auto-commit structural changes. |
| **Cascade-Regeneration Worker** | The background agent triggered by hard-delete and quarantine events. Walks affected claims and regenerates affected pages atomically, with budget caps. |
| **Soft delete** | Source marked deleted; hidden from retrieval; claims citing it flagged for the next lint pass; audit events untouched. |
| **Hard delete** | Source and all derivatives permanently removed; affected wiki pages regenerated; audit events tombstoned (metadata kept, content references scrubbed). |
| **Quarantine** | Source removed from active retrieval but retained in cold storage; pages flagged but not regenerated until the source's disposition is decided (typically by Legal). |
| **Right-to-be-forgotten (RTBF)** | The user/legal-initiated request to purge content. Implemented through hard delete (single source) or the concept-level RTBF flow (cross-corpus search and bulk delete). |
| **Editorial filter** | The LLM-scored stage of ingestion that compares a source against the department's policy and decides accept / reject / route-to-queue. |
| **Audit ledger** | The append-only `audit_events` table partitioned by month. SOC2-grade record of every meaningful action. Exported to Microsoft Sentinel. |
| **LLM Gateway** | The .NET service (built on Microsoft Semantic Kernel) through which all LLM calls in the system flow. Provider-agnostic; emits per-call telemetry to the audit ledger. |
| **RLS** | Row-Level Security in PostgreSQL. The mechanism by which department isolation is enforced at the database level. |
| **Classification** | A four-tier label on every source: `Public`, `Internal`, `Confidential`, `Restricted`. **The default access boundary**: `Internal` (default) is readable by any authenticated employee, `Confidential` is gated to the owning department, `Restricted` to the owning department's `Librarian`+. Also drives approval rules, retention, audit filtering, and a UI badge. See [ADR 0011](adr/0011-data-classification.md). |
| **Source share** | An explicit row in `source_shares` granting a specific department read access to a single `Confidential` or `Restricted` source. Granted by the source's owning Librarian (or Admin for `Restricted`); auditable, revocable, optionally time-bounded. The escape hatch when classification + department alone don't fit — e.g., Finance shares the FY27 budget doc with Engineering for project Atlas planning. |
| **Visibility (read access)** | The combined effect of classification + department membership + source shares. Distinct from **authority** (who can write / approve / govern / delete), which is gated strictly by role within the owning department regardless of classification. **Persona is not a visibility dimension** — see ADR 0005's amendment. |
| **Persona** | A defined work-context (Engineering, Product, SRE, Sales, Marketing, Customer Success, Legal, HR). The fourth organizing dimension alongside department, role, and classification. A persona shapes *what the system retrieves*, *how the system synthesizes the answer*, and *which internal autonomous actions the user may invoke*. A persona does **not** shape what the user can see — visibility stays governed by classification + department + source shares. See [ADR 0014](adr/0014-personas-first-class.md). |
| **Persona membership** | A user's grant of a persona, optionally department-scoped and optionally time-bounded. A user may hold multiple persona memberships simultaneously; each session designates a *primary persona* that drives retrieval and synthesis. Granted by Admin (system-wide) or department Librarian (within their department's scope). |
| **Persona-aware retrieval** | The retrieval pipeline is parameterized by the active persona: source-type weights, recency curve, authority bias, and (optionally) classification floor are persona-specific. Persona-aware reranking runs *after* RLS filtering, so persona never widens the authorized result set. See [ADR 0015](adr/0015-persona-aware-retrieval-synthesis.md). |
| **Persona facet** | A page facet keyed by `(min_classification, persona_id)`. A wiki page may have a persona-neutral facet plus persona-shaped facets when the source pool warrants. The reader's primary persona selects which facet they see by default; classification still gates visibility. |
| **Persona action** | An internal action a persona may invoke (classify ticket, route ticket, attach runbook, draft summary, flag at-risk account, etc.). Each persona has a documented `default_action_set`. Carve-outs are permanent: no autonomous customer-facing actions, and no AI-direct money/refund decisions. See [ADR 0016](adr/0016-persona-internal-autonomous-actions.md). |
| **Recommend mode** | An action mode where the system surfaces a suggestion for a human; the human decides. The default mode for any new action; a baseline operating mode for many actions indefinitely. |
| **Shadow mode** | An action mode where the system *simulates* a decision and logs what it *would have* done, without real-world effect. The intermediate stage between Recommend and Autonomous; used to accumulate evidence of agreement-with-human. |
| **Autonomy gate** | The set of evidence thresholds an action must meet to advance from Recommend → Shadow or from Shadow → Autonomous. Gates require ≥30 days in the prior mode, ≥200 (Recommend) or ≥500 (Shadow) evaluated decisions, ≥85% (Recommend) or ≥90% (Shadow) agreement-with-human, plus reversibility validation and Sponsoring Persona Owner sign-off. |
| **Reversal path** | The documented mechanism to undo an autonomous-mode action's effect (e.g., re-classify, re-route, restore prior priority, untag, un-merge cluster). Reversibility-by-design is required for any action that runs in non-Recommend mode; actions whose effects are not cleanly reversible are not Autonomous candidates. |
| **Sponsoring Persona Owner** | The named human accountable for a persona's action set. Approves additions, signs off on Recommend → Shadow promotions, reviews persona-quality dashboards, and owns reversibility-path realism review. Per persona, named in `docs/personas/<name>.md`. |
